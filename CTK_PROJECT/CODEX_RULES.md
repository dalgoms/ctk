# CTK ChoiceDX Codex Rules

## 목적

### Why
CTK ChoiceDX 프로젝트는 브랜드, SEO/GEO, 홈페이지, UX, 세일즈, PR, SNS, 문서화가 서로 연결되어 있다. 따라서 Codex가 매번 다른 방식으로 작업하면 프로젝트 전체 일관성이 깨질 수 있다.

### What
이 문서는 앞으로 CTK ChoiceDX 프로젝트에서 Codex가 항상 따라야 하는 작업 규칙이다.

### How
모든 분석, 제안, 문서, 실행 가이드, 리뷰, 산출물은 아래 규칙을 따른다.

### Result
프로젝트의 모든 결과물이 분석 기반, IA 기반, SEO/GEO 기반, 재사용 가능한 형태로 유지된다.

---

## Core Rules

### 1. 실행보다 분석을 먼저 한다.
- 어떤 작업이든 바로 만들지 않는다.
- 먼저 현재 상황, 문제, 원인, 필요한 자료를 분석한다.
- 분석 없이 나온 실행안은 초안으로만 본다.

### 2. 무조건 IA를 먼저 만든다.
- 홈페이지, 콘텐츠, UX, 세일즈 자료, PR, SNS 모두 정보 구조를 먼저 정의한다.
- IA는 고객이 이해해야 하는 순서를 기준으로 만든다.
- IA 없이 디자인, 카피, 화면 리뷰를 시작하지 않는다.

### 3. 바로 디자인하지 않는다.
- Codex는 디자인 실작업자가 아니다.
- 디자인 전에 목적, 사용자, 메시지, IA, 화면별 역할을 먼저 정리한다.
- Figma Review는 디자인 취향이 아니라 UX, 메시지, 전환, KPI 기준으로 한다.

### 4. 항상 Why부터 설명한다.
- 모든 설명은 Why에서 시작한다.
- 기본 구조는 `Why → What → How → Result`를 따른다.
- 실무 문서에서는 `문제 → 원인 → 해결 → 실행 → 성과` 구조를 유지한다.

### 5. 항상 경쟁사와 비교한다.
- ChoiceDX만 단독으로 판단하지 않는다.
- BECON, ARAM Huvis, 기타 직접/간접 경쟁사와 비교한다.
- 비교 기준은 Brand, Website, SEO, AI Search, Homepage, UX, SNS, Product, Sales, Content, PR, Keyword, Positioning, Technology, Trust, Global을 포함한다.

### 6. SEO와 GEO를 동시에 고려한다.
- Google 검색 노출만 보지 않는다.
- ChatGPT, Gemini, Claude, Perplexity 등 AI Search에서의 발견 가능성과 설명 가능성을 함께 본다.
- Metadata, Keyword, Entity, Citation, Authority, FAQ, Structured Data를 함께 점검한다.

### 7. 모든 결과물은 문서화한다.
- 결정 사항, 분석 결과, 실행 기준, 검수 결과, 다음 작업을 남긴다.
- 구두로 정리된 내용도 문서화 대상이다.
- 문서에는 목적, 담당, 완료조건, 필요한 자료, 관련 Task를 최대한 포함한다.

### 8. 항상 다음 Action을 제안한다.
- 모든 분석과 산출물은 다음 행동으로 끝난다.
- 다음 Action은 Priority, Owner, Due, Dependency, Expected Output과 연결한다.
- 막힌 항목은 Blocked로 표시한다.

### 9. 모든 Deliverable은 재사용 가능하게 만든다.
- 내부 담당자가 이어받아 반복 사용할 수 있어야 한다.
- 일회성 보고서보다 템플릿, 체크리스트, 가이드 형태를 우선한다.
- 문서는 다른 산출물과 연결 가능한 구조로 만든다.

### 10. 프로젝트 전체 일관성을 항상 유지한다.
- Project Context, Guide, Workflow, Execution Plan, Task Backlog, Deliverables, Dashboard와 충돌하지 않게 작업한다.
- 메시지, IA, SEO/GEO, UX, 세일즈, PR, SNS가 같은 방향을 유지해야 한다.
- 새 산출물이 생기면 기존 문서와 연결한다.

## Default Work Order

### Why
작업 순서가 흔들리면 실행이 분석을 앞서고, 산출물이 재작업될 가능성이 높아진다.

### What
Codex는 모든 요청을 아래 순서로 처리한다.

### How
1. 현황 분석
2. 경쟁사 비교
3. SEO/GEO 관점 확인
4. IA 또는 정보 구조 정의
5. 우선순위 설정
6. 실행안 작성
7. 검수 기준 정리
8. 문서화
9. 다음 Action 제안

### Result
모든 작업이 프로젝트 운영 원칙과 같은 흐름으로 진행된다.

---

## Default Output Structure

### Why
설명 방식이 매번 달라지면 내부 공유와 의사결정이 느려진다.

### What
기본 출력 구조를 고정한다.

### How
```markdown
## [Task Name]

### Why
-

### What
-

### How
-

### Result
-

### Next Action
-
```

또는 실무 진단 문서에서는 아래 구조를 사용한다.

```markdown
## [Task Name]

### 문제
-

### 원인
-

### 해결
-

### 실행
-

### 성과
-

### Next Action
-
```

### Result
결과물이 항상 같은 문법으로 읽히고 재사용 가능해진다.

## Review Checklist

| Check | Question | Status |
|---|---|---|
| Analysis First | 분석 없이 실행하지 않았는가 |  |
| IA First | IA 또는 정보 구조가 먼저 정의되었는가 |  |
| No Direct Design | 디자인 실작업 전에 목적과 구조가 정리되었는가 |  |
| Why First | Why부터 설명했는가 |  |
| Competitor Comparison | 경쟁사와 비교했는가 |  |
| SEO + GEO | SEO와 GEO를 동시에 고려했는가 |  |
| Documentation | 결과물을 문서화했는가 |  |
| Next Action | 다음 Action을 제안했는가 |  |
| Reusable Deliverable | 산출물이 재사용 가능한가 |  |
| Project Consistency | 기존 프로젝트 문서와 일관적인가 |  |
